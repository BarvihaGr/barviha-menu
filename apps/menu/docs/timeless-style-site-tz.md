# Техническое задание на разработку сайта (концепт Timeless.club) — для «Арка»

Дополняет [arka-menu-tz.md](arka-menu-tz.md) (там — токены/компоненты/код).
Здесь — концепт, подача и стилистика уровня продукта/сайта в целом, разобранные
на живом аудите timeless.club (структура страницы, CSS-бандл, поведение).

---

## 1. Концепция и цели проекта

Сайт-меню лаунж-формата (по образцу Timeless), транслирующий атмосферу бренда
через **сдержанную роскошь**, а не через изобилие визуальных эффектов:

- **Один акцент — не палитра.** У Timeless весь цвет держится на одном оттенке
  (`#bc867d`, приглушённая терракота) поверх тёмного/светлого нейтрала. Никаких
  «ярких» вторичных цветов. У Арки то же самое: единственный акцент —
  бронза-золото `#C49262` на тёплом почти-чёрном фоне.
- **Типографика — главный инструмент, не декор.** Заголовки капсом, крупная
  шкала (24–100px), широкий трекинг на лейблах (0.05–0.07em) — иерархия
  строится шрифтом и отступом, а не рамками/тенями/градиентами.
- **Ничего лишнего между гостем и меню/бронью.** Хедер — лого + локация/язык +
  «Резерв». Никакого второго уровня меню, никаких баннеров/попапов при входе.
- **Цель:** удобная навигация по меню (кальяны/бар/кухня → секции → позиции) +
  лёгкий вход в бронирование стола, без ощущения «сайта-каталога».

## 2. Анализ функционала (на примере Timeless.club)

| Функция | Как сделано у Timeless | Как переносим на Арку |
|---|---|---|
| **Мультилокация** | Переключатель город: Москва / Дубай — в хедере, рядом с языком | У нас уже шире: переключатель локаций по всей сети (сейчас в проде — Арка/Киевская, инфраструктура рассчитана на 27+ точек по городам, см. `location-theme.ts`) |
| **Версионность (языки)** | EN/RU-переключатель в хедере | У нас уже 4 локали: ru/en/zh/hy (`i18n/routing`) — шире, чем у референса |
| **Интерактивное меню** | Разделы: Бар / Ароматы (кальяны) / Закуски / Сезонное предложение. Внутри раздела — под-категории (Классика/Авторские/Фруктовые), sticky-навигация, continuous-scroll с scroll-spy | 1:1 наш паттерн в `CoffeeMenuList`: секции по `item.sub`, sticky scroll-spy чипы — уже реализовано, нужно добавить верхний таб-бар Кальяны/Бар/Кухня в той же стилистике (капс, пилюли, счётчик позиции) |
| **Карточки товаров** | Фото, название капсом, цена, описание вкуса/состава. **Вариации** — отдельные строки-позиции внутри одной карточки-группы (Б/А версия, просекко, шампанское — как отдельные строки с разной ценой под одним названием) | Реализовать через существующее поле `sub`/группировку: одна «родительская» позиция → несколько ценовых строк-вариаций (объём/версия), а не отдельные карточки — это новый паттерн, отсутствует в текущей `CoffeeItemCard` (сейчас 1 карточка = 1 фиксированная цена) |
| **Бронирование** | Кнопка «Резерв» в хедере (везде, sticky) → модалка с календарём (выбор даты) и, вероятно, формой/интеграцией с системой бронирования | Нужно завести: кнопка «Резерв» в `CoffeeHeader` рядом с бургером → модалка (дата/время/кол-во гостей/телефон) → отправка в Telegram/CRM (см. интеграции ниже) |

## 3. Структура сайта (Sitemap)

```
/{locale}/arka
├── Главная            — hero (видео/фото + лого), быстрые ссылки на разделы меню,
│                         афиша ближайших событий (уже есть данные — AFISHA_BY_SLUG)
├── Меню
│   ├── Кухня           — Категория → под-категория (sub) → позиции
│   ├── Бар             — то же дерево, включая вариации (Б/А, объёмы)
│   └── Кальяны         — ароматы/миксы, hero-позиция (premium) + остальные
├── Локации             — описание точки, карта, контакты, режим работы
├── Новости / Афиша      — DJ-сеты, дегустации, тематические вечера
│                         (контент уже готов на уровне данных — Spotlight/Afisha)
└── Резерв              — модалка/форма бронирования (дата, время, гости, телефон)
```

Дерево категорий меню (Категория → Подкатегория → Позиция) уже реализовано
инфраструктурно (`CoffeeMenuList` + `item.sub`) — задача не «построить с нуля»,
а «докрутить подачу» (верхний таб-бар разделов, вариации в карточке, модалка
резерва).

## 4. Технические требования

### Дизайн и Frontend
- **Премиальный минимализм, typography-driven** — шкала/трекинг/капс как в
  §1 [arka-menu-tz.md](arka-menu-tz.md), один акцентный цвет на весь сайт локации.
- **Mobile-first, безупречно на телефоне** — меню это то, что открывают за
  столом на телефоне гостя: приоритет читаемости карточки в одну руку,
  крупная зона тапа на «добавить»/«вариация», sticky-навигация не съедает
  экран (высота хедера уже параметризована токеном `--header-height`).
- **Плавное появление элементов** — уже подключено (`framer-motion` в
  `CoffeeItemCard`: fade+slide при входе во вьюпорт). Добавить по референсу
  `clip-path`-переход при смене раздела/секции меню (см. §1.4
  arka-menu-tz.md) — это то, что у Timeless считывается как «дорого», а не
  обычный fade.

### Backend и данные
- У нас это уже не «CMS общего назначения» (Strapi/Webflow/WordPress+ACF из
  типового ТЗ), а собственный слой `@barviha/db` (`packages/db`) с типизированной
  моделью позиций/цен/локаций (`GEN_ITEMS`, `prices[slug]`, `ResolvedMenuItem`).
  Для менеджеров, которые должны сами редактировать цены/описания без
  участия разработчика, — нужна тонкая админ-панель поверх этого слоя
  (CRUD по ценам/составу/фото/вариациям), а не внедрение стороннего Strapi —
  дешевле и не дублирует уже написанную типизацию.
- **Вариации товара** (объём/Б-А/просекко-шампанское) — расширение модели
  позиции: `variants: { label, price }[]` вместо одного `price`. Нужна
  миграция схемы (`packages/db`) и точечное изменение `CoffeeItemCard`/страницы
  позиции под рендер списка вариаций вместо одной цены.
- **Производительность** — `next/image` уже используется (оптимизация +
  автоматический lazy loading вне viewport), это закрыто по умолчанию;
  дозаливка недостающих фото блюд (см. §4 arka-menu-tz.md — кухня Арки 8/92
  с фото) — контентная задача, не техническая.

### Бронирование (не было в предыдущем ТЗ, здесь — новый контур)
- Кнопка «Резерв» — постоянно видима (в `CoffeeHeader`, sticky).
- Модалка: дата (календарь), время, кол-во гостей, телефон/имя.
- Приёмник заявки — Telegram-бот/канал ресторана (проще и быстрее чем
  внешний виджет-интеграция; соответствует остальной архитектуре проекта,
  где уведомления уже завязаны на Telegram) либо форма-вебхук в CRM, если
  она уже используется сетью — уточнить у пользователя перед реализацией.

## 5. Этапы разработки

1. **Прототип/UX меню-разделов** — верхний таб-бар Кальяны/Бар/Кухня в стиле
   Timeless (капс, пилюли, индикатор позиции) поверх уже готового
   `CoffeeMenuList`.
2. **UI-полировка бренд-эстетики Арки** — `clip-path`-переход между
   разделами/секциями, финальная выверка контраста бронза-палитры (уже
   заведена в `coffee-design.ts`).
3. **Вариации товара** — модель данных + карточка/страница позиции с
   несколькими ценовыми строками под одним названием.
4. **Резерв** — кнопка в хедере + модалка + приёмник заявки.
5. **Афиша/Новости на витрине** — вывести уже существующие данные
   (`AFISHA_BY_SLUG`) блоком на главной Арки.
6. **QA на реальном телефоне** + прогон `next build` перед деплоем (см.
   memory: dev-режим не ловит ошибки типов — обязательный build-check).

Пункты 1–2 логически продолжают уже сделанное (палитра/острые углы/шрифт
подключены и проверены на localhost:3000, см. предыдущий ответ). Готов начать
с любого пункта — по умолчанию предлагаю дальше делать **п.1 (таб-бар
разделов)**, так как он самый заметный визуально и не требует изменения схемы
данных.
